# **產品需求文件 (PRD)：行銷公司行動版報價單產生器**

## **1. 產品概述**

本專案旨在為行銷公司業務團隊開發一款「內部報價單自動生成工具」。考慮到業務人員經常在外奔波，系統需高度優化手機端 (Mobile-first) 的操作體驗。透過整合預建的商品資料與客戶名單，並自動處理稅務與折扣計算，最終產出帶有企業識別的專業 PDF 報價單，以大幅降低行政作業時間。

## **2. 開發目標 (User Story)**

* **自動帶入省時間**：身為行銷公司業務，我希望可以選擇廠商及商品就能自動產生報價單，這樣就不需要耗費找資料及複製貼上的時間。  
* **客戶資料留存**：身為行銷公司業務，我希望初次開報價單的廠商資訊可以被存下來，以後可以直接選擇廠商來產生報價單，節省重複輸入的時間。  
* **商品資料庫**：身為行銷公司業務，我希望系統有一個「商品資料庫」可以事先建檔（包含品名、規格、預設單價），這樣我在開報價單時才能直接用關鍵字或下拉選單帶入，不用每次手打商品名稱與價格。  
* **防篡改匯出**：身為行銷公司業務，我希望產生的報價單可以被匯出成 PDF，才不會容易被改到資料。  
* **稅額自動計算**：身為行銷公司業務，我希望系統能自動計算營業稅（例如：可切換含稅/未稅、外加 5% 稅率），這樣我就不用拿計算機在那邊自己按，也不會算錯總金額。  
* **彈性折扣設定**：身為行銷公司業務，我需要在製作報價單時，可以用「%數」或「固定金額」來設定折扣，讓我有彈性議價的空間，也不用自己計算總金額。  
* **專業品牌形象**：身為行銷公司業務，我希望匯出的 PDF 報價單上，能自動帶入我們公司的 Logo、公司統編，以及我個人的聯絡資訊（姓名、Email、電話號碼），讓報價單看起來更專業且一致。  
* **確認締約 (新增)**：身為行銷公司業務，我希望匯出的報價單底部有專屬的「客戶簽名回傳區塊」，方便客戶確認後直接簽名（或蓋公司章）回傳，完成締約程序。

## **3. 使用流程 (User Flow)**

考量手機操作，流程設計為線性的步驟：

1. **身分選擇**：開啟網頁後，首先從下拉選單選擇「自己（業務員）」，系統會在背景帶入其姓名、Email、電話號碼，用於最終生成的報價單。  
2. **填寫/選擇報價對象**：  
   * 提供「搜尋現有廠商」的下拉選單或自動完成輸入框。  
   * 若為新廠商，業務直接填寫公司名稱、統編、聯絡人等資訊，系統會在送出報價單時，於背景將新客戶寫入資料庫。  
3. **建立報價明細 (核心操作)**：  
   * 點擊「新增品項」，彈出/展開商品搜尋框。  
   * 透過關鍵字或分類選單找到商品，點擊後帶入預設品名、規格與單價。  
   * 業務可微調該品項的「數量」或臨時修改「單價」。  
   * 重複此步驟加入多個品項。  
4. **設定交易條件與金額微調**：  
   * 設定稅額：切換 [不計稅 / 外加 5% 營業稅 / 已含稅]。  
   * 設定折扣：輸入折扣數值，並切換是 [折讓金額 ($)] 還是 [折扣趴數 (%)]。  
   * 系統即時顯示小計、折扣、稅額、最終總計。  
5. **預覽與匯出 PDF**：  
   * 點擊「預覽報價單」，畫面轉換為 A4 比例的唯讀預覽模式（包含公司 Logo、業務資訊與**底部客戶回簽區塊**）。  
   * 確認無誤後，點擊「匯出 PDF」下載檔案至手機/電腦，同時觸發「儲存新客戶(若有)」的背景 API。

## **4. 技術棧 (Tech Stack)**

* **平台**：Web Application (SPA)。  
* **前端**：建議使用 React (Hook 為主) + Tailwind CSS (方便快速切版與實現 Mobile-first)。使用原生瀏覽器列印 API 或 html2pdf.js 等輕量套件產出 PDF。前端需編譯為靜態檔案。  
* **部署**：前端靜態檔案部署於 **GitHub Pages** (因此前端不能有複雜的伺服器依賴，也不使用需要複雜運算的動畫套件)。  
* **後端 / API**：**Google Apps Script (GAS)** 部署為 Web App (提供 GET/POST API)。  
* **資料庫**：**Google Sheets**。

## **5. 資料庫結構 (Google Sheets 規劃)**

預計使用同一個 Google Spreadsheet，包含以下三個 Sheet：

### **Sheet 1: 廠商資訊 (Vendors)**

| 欄位名稱 | 說明 |
| :---- | :---- |
| id | 唯一碼 (如 UUID 或 Timestamp) |
| companyName | 公司名稱 (必填) |
| taxId | 統一編號 |
| contactPerson | 聯絡人姓名 |
| phone | 聯絡電話 |
| email | 聯絡信箱 |

### **Sheet 2: 商品資訊 (Products)**

| 欄位名稱 | 說明 |
| :---- | :---- |
| id | 唯一碼 |
| name | 商品品名 (必填) |
| specification | 規格說明/細節描述 |
| defaultPrice | 預設單價 (數字) |

### **Sheet 3: 業務員資訊 (SalesReps)**

| 欄位名稱 | 說明 |
| :---- | :---- |
| id | 唯一碼 |
| name | 業務員姓名 (必填) |
| email | 業務員信箱 |
| phone | 電話號碼 |

## **6. 畫面規劃與 UI/UX 限制**

* **Mobile-First 優先**：  
  * 由於業務多用手機操作，**禁止使用在手機上需左右滑動的傳統寬大表格 (Table)** 來呈現商品明細。  
  * 明細應改為「卡片式 (Card) 堆疊」設計。每個品項是一張卡片，內含品名、數量(加減按鈕)、單價輸入框。  
* **效能與限制**：  
  * 配合 GitHub Pages，不使用龐大的動畫庫 (如 Framer Motion 等)，過渡效果使用純 CSS (Tailwind 的 transition) 即可。  
  * 輸入區域應盡量使用大按鈕 (Touch Target > 44px)，方便手指點擊。  
  * 常駐底部的浮動操作列 (Sticky Bottom Bar) 顯示「目前總金額」與「匯出 PDF 按鈕」，讓業務隨時知道報價進度。

## **7. 完成標準 (Acceptance Criteria)**

1. 畫面能在手機端順暢操作，輸入資料不破版。  
2. 能成功從假資料 (後續為 Google Sheet) 載入下拉選單 (業務、廠商、商品)。  
3. 能正確新增/刪除報價品項，並自動計算單列小計。  
4. 能正確應用 5% 稅額邏輯與 折扣邏輯 (支援 % 與固定金額)，算出最終總計。  
5. 能成功將當前畫面轉換並匯出為具備公司 Logo、統一編號、業務資訊與**客戶回簽區塊**的 A4 格式 PDF。  
6. (對接後端時) 能成功觸發 POST 請求將未知的新廠商寫入 Google Sheet。

## **8. 不做什麼 (Out of Scope - 非常重要)**

* **不開發**：紀錄並儲存「過去開過的每一張報價單明細」功能。  
* **不開發**：叫出舊的報價單並「複製過去報價單」的需求。  
* **不開發**：複雜的使用者登入/身分驗證系統 (目前只需下拉選擇業務員身分即可)。

## **9. 執行計畫 (開發指導方針)**

1. **Phase 1: 前端 UI 與邏輯開發 (目前階段)**  
   * 全面使用 Mock Data (假資料 JSON) 來模擬 Google Sheets 的回傳。  
   * 實作完整的手機端 UI 切版。  
   * 實作所有價格計算邏輯 (單價、數量、稅額、折扣)。  
   * 實作 PDF 預覽與匯出功能。  
2. **Phase 2: 後端串接 (下一階段)**  
   * 撰寫 Google Apps Script 讀寫 Sheet。  
   * 將前端 Mock Data 替換為 fetch() 呼叫 GAS API。  
3. **溝通原則**：  
   * 作為 AI 開發助手，在撰寫任一階段的程式碼之前，如果發現 PRD 有任何邏輯衝突或技術上不可行之處（尤其是 PDF 套件的選擇或手機版排版的細節），**請務必先停下來向我提問釐清，確認後再進行代碼生成**。
